        \vspace{0.4cm}
        
        \begin{tikzpicture}[

\begin{frame}
    \begin{center}
        \textcolor{FooterGreen}{\Large \textbf{Avoid SQL request in loop (GCI72)}}
        \vspace{0.4cm}
        
        \begin{tikzpicture}[
            box/.style={draw=FooterGreen, thick, rounded corners, text=white, align=left, fill=BgGreen!50!black, font=\scriptsize, text width=12.5cm, inner sep=0.2cm},
            arrow/.style={->, thick, FooterGreen, >=Latex}
        ]
            
            \node[box] (problema) {
                \textbf{Problema} \\[1ex]
                Query SQL in cicli iterativi generano enorme overhead DBMS, bloccando i thread Tomcat e impedendo il basso consumo della CPU. Inefficienze rilevate:\\[0.5ex]
                $\bullet$ \textbf{Assenza Batch Processing}: singole \texttt{INSERT/UPDATE} in ciclo forzano transazioni e commit continui per ogni record, frammentando l'I/O su disco e saturando la rete.\\
                $\bullet$ \textbf{Problema N+1}: recuperare \textit{N} ordini e i loro prodotti costa \textit{1+N} query distinte (1 per ordini, \textit{N} per prodotti), moltiplicando roundtrip e attese I/O.
            };
            
            \node[box, below=0.3cm of problema] (soluzione) {
                \textbf{Soluzione} \\[1ex]
                $\bullet$ \textbf{Risoluzione Scritture}: introdotte operazioni \textbf{batch native JDBC} (\texttt{addBatch()}, \texttt{executeBatch()}). I comandi accodati vengono inviati in un singolo pacchetto di rete ed eseguiti in un'unica transazione atomica, abbattendo lock e costi di connessione.\\[0.5ex]
                $\bullet$ \textbf{Risoluzione N+1}: le \textit{1+N} query sono state ridotte a \textbf{1 query aggregata} (\texttt{LEFT JOIN}) in un solo roundtrip. L'assemblaggio senza duplicati è delegato alla RAM tramite \texttt{LinkedHashMap}, annullando del tutto gli sprechi di rete.
            };
            
            \draw[arrow] (problema) -- (soluzione);
            
        \end{tikzpicture}
    \end{center}
\end{frame}

\begin{frame}
    \begin{center}
        \textcolor{FooterGreen}{\Large \textbf{Do not call a function when declaring a for-type loop (GCI69)}}
        \vspace{0.4cm}
        
        \begin{tikzpicture}[
            box/.style={draw=FooterGreen, thick, rounded corners, text=white, align=left, fill=BgGreen!50!black, font=\footnotesize, text width=12cm, inner sep=0.25cm},
            arrow/.style={->, thick, FooterGreen, >=Latex}
        ]
            
            \node[box] (problema) {
                \textbf{Problema} \\[1ex]
                Invocare funzioni \textbf{invarianti} (il cui risultato non cambia mai durante l'esecuzione) all'interno dei parametri di dichiarazione o di guardia di un ciclo forza la CPU a ricalcolare l'intero metodo ad ogni singola iterazione, causando un enorme e inutile dispendio di calcolo.
            };
            
            \node[box, below=0.6cm of problema] (soluzione) {
                \textbf{Soluzione} \\[1ex]
                I risultati di queste funzioni sono stati estratti e salvati in \textbf{variabili primitive locali} \textit{prima} dell'inizio del ciclo.\\[1ex]
                Ciò elimina alla radice migliaia di invocazioni ridondanti, fungendo da vero e proprio \textit{caching} preventivo e consentendo alla JVM di ottimizzare pienamente l'esecuzione del blocco iterativo.
            };
            
            \draw[arrow] (problema) -- (soluzione);
            
        \end{tikzpicture}
    \end{center}
\end{frame}

\begin{frame}
    \begin{center}
        \textcolor{FooterGreen}{\Large \textbf{Don't use the query SELECT * FROM (GCI74)}}
        \vspace{0.4cm}
        
        \begin{tikzpicture}[
            box/.style={draw=FooterGreen, thick, rounded corners, text=white, align=left, fill=BgGreen!50!black, font=\footnotesize, text width=12cm, inner sep=0.25cm},
            arrow/.style={->, thick, FooterGreen, >=Latex}
        ]
            
            \node[box] (problema) {
                \textbf{Problema} \\[1ex]
                L'uso sistematico di \texttt{SELECT *} costringe il database a pesanti operazioni di I/O su disco per estrarre \textbf{tutti i campi}, inclusi quelli inutili o enormi (es. immagini in \textit{Base64}). Ciò genera traffico di rete sprecato e satura la JVM con massicce allocazioni di oggetti temporanei.
            };
            
            \node[box, below=0.6cm of problema] (soluzione) {
                \textbf{Soluzione} \\[1ex]
                Le query sono state riscritte esplicitando nella proiezione SQL \textbf{unicamente le colonne necessarie}.\\[1ex]
                Il taglio del \textit{payload} di rete alleggerisce la fase di mappatura ad oggetti (ORM), abbattendo le allocazioni nella memoria Heap e minimizzando l'intenso stress energetico causato dai cicli continui di \textbf{Garbage Collection}. L'allineamento dei risultati ai nomi simbolici delle colonne aumenta inoltre la resilienza strutturale del codice.
            };
            
            \draw[arrow] (problema) -- (soluzione);
            
        \end{tikzpicture}
    \end{center}
\end{frame}

                Solo quando il codice è stato completamente ottimizzato dal JIT, lo script lancia le \textbf{5 iterazioni di misurazione reale} a singolo thread, registrando il vero costo energetico dell'algoritmo a regime.

                    $\bullet$ \textbf{Durata Totale ($T_{global}$, s):} L'esatta finestra di simulazione:
                    \[ T_{global} = \sum_{k=1}^{M} \Delta t_k \]
                    
                    $\bullet$ \textbf{Energia CPU Package ($E_{CPU}$, J):} Il costo energetico "netto" dell'operazione (inclusi \textit{core}, \textit{cache} e controller):
                    \[ E_{CPU} = \sum_{k=1}^{M} \Delta E_{CPU, k} \]
                    
                    $\bullet$ \textbf{Energia Core ($E_{Core}$, J):} L'assorbimento limitato ai soli \textit{core}. È l'indicatore principe dell'efficienza algoritmica:
                    \[ E_{Core} = \sum_{k=1}^{M} \Delta E_{Core, k} \]
                \end{minipage}%
                \hfill
                \begin{minipage}[t]{0.48\textwidth}
                    \vspace{0pt}
                    $\bullet$ \textbf{Potenza Media ($P_{avg}$, W):} Il tasso medio di consumo. Un abbassamento indica maggiore permanenza nei \textit{C-states}:
                    \[ P_{avg} = \frac{E_{CPU}}{T_{global}} \]
                    
                    $\bullet$ \textbf{CPU Usage \& Freq (\% e MHz):} Medie e picchi. Algoritmi snelli abbassano la frequenza operativa e il voltaggio, smorzando il calore e l'\textit{active waiting} (\textit{spin-locks}).\\[2ex]
                    
                    $\bullet$ \textbf{RAM Usage (Media e Picco, MB):} L'impronta allocata nel sistema. Contenere la RAM nella JVM è l'unica via per scongiurare onerosi cicli di \textit{Garbage Collection}.
                \end{minipage}
