Your goal is to generate a COMPLETE, idiomatic JUnit 5 test class that strictly implements the test frames defined in the report.

================================================================================
STRICT CONSTRAINTS (DO NOT IGNORE)
================================================================================

1. FRAME ADHERENCE
   - Do NOT re-analyze the domain logic. Trust the Report.
   - You MUST implement every Test Frame (e.g., F1, F2, etc.) listed in the report.
   - Map frames to test methods one-to-one (or grouped via ParameterizedTest).

2. JUNIT 5 EXCLUSIVE
   - Use ONLY JUnit 5 (Jupiter) annotations: @Test, @BeforeEach, @DisplayName, @ParameterizedTest, @CsvSource.
   - Do NOT use JUnit 4 (e.g., @Rule, @Before).
   - Use org.junit.jupiter.api.Assertions (assertAll, assertEquals, assertThrows).

3. NAMING AND LOCATION
   - Class Name: [TargetClass]Test
   - Package: Must match the package of the target class.
   - Methods: Use descriptive names (e.g., `testAggiungiRecensione_ValidData_Success`).

4. COLLABORATORS (NO MOCKING PREFERENCE)
   - Do NOT use Mockito unless the collaborator is an interface or complex external service.
   - Prefer creating private helper methods to instantiate valid concrete objects (e.g., `private Utente createValidUser()`).

================================================================================
CODING STYLE & STRUCTURE
================================================================================

[STRUCTURE]
- Use the AAA pattern (Arrange, Act, Assert) with comments in every method:
  // Arrange
  ...
  // Act
  ...
  // Assert
  ...

[ASSERTIONS]
- When verifying object state, use assertions.assertAll() to group checks.
- When expecting errors, use assertions.assertThrows().

[PARAMETERIZED TESTS]
- If the report groups frames effectively (e.g., "Frames 1, 2, 3 are valid boundaries"), implement them as a @ParameterizedTest using @CsvSource.
- Use the "name" attribute in @ParameterizedTest to make execution logs readable.
